Prototype Specifications

Between a reference architecture and a finished system sits the part most transformations skip: an executable specification, proven before anyone commits to build. This is how we turn a pattern into something a bank can hand to any vendor with confidence.

From pattern to running system

Five steps, each a distinct artefact with its own status.

Reference Architecture→ Prototype Specification→ Working Prototype→ Working Specification→ Executable System

What each one means

Reference Architecture

A reusable pattern for a class of problem. C3 is ours for governed human-machine action.

Prototype Specification

The executable blueprint produced from a reference architecture and a journey: precise enough to build from.

Working Prototype

A running proof of that specification on synthetic data: our demonstration.

Working Specification

The portable, standards-based artefact a client takes to whatever vendor they choose. Vendor-neutral, no lock-in.

Executable System

What the chosen vendor builds and the bank runs in production. The client owns it.

Why it matters

Most banks jump from a slide to a build and discover the hard parts in production. A Prototype Specification proves them first, expressed in open standards (ISO 20022, BIAN, recognised control frameworks) so it is portable. It also carries a performance envelope: where the design holds, and where it breaks, as non-functional requirements a vendor must meet.

What runs today is the working prototype, a demonstration on synthetic data. Prototype Specifications are labelled Specified or Blueprint until implementation evidence supports promotion to Demonstrated.
