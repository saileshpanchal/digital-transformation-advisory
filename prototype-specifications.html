Prototype Specifications

A Prototype Specification is the executable blueprint produced from a reference architecture and a journey. It is how a decision moves from a pattern to something a client can build, expressed in open standards so it stays portable and vendor-neutral.

Where a specification sits

From the mission down to a system a client runs. A reference architecture and a journey produce a Prototype Specification, proven in a working model, carried out as a working specification, and built into an executable system.

Five terms, each with one job

We keep these distinct so a client always knows what they are looking at.

Reference architecture

A reusable pattern for a recurring problem, such as C3. It is not specific to one bank or one journey.

Prototype specification

The executable blueprint produced from a reference architecture and a journey. The bridge from pattern to build.

Working prototype

A running proof on Accountable Bank, on synthetic data, with test consumers and never real customers.

Working specification

The portable artefact a client takes to any vendor, expressed in ISO 20022, BIAN, and recognised control frameworks.

Executable system

What the client builds and runs from the specification, with their chosen vendor and stack.

Published examples, with status and provenance

Each carries an honest label. A demonstration genuinely runs; a blueprint is designed and specified, not yet demonstrated end to end.
